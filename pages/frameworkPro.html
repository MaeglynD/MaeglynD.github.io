    SASS & KAITEN

Sass is a fantastic preprocessor, popular amongst most front end developers for speeding up the entire styling process. Using it has been pretty fun, and to showcase some of sass’s features i decided to create a fake cryptocurrency startup site called ‘Kaiten’. In my eyes, the three main features of sass that stood out were; nesting capabilities, variables and mixins. Nesting with sass is easy, simply put the element styles within the parent element’s styles. This becomes useful when you’re working with multiple stylesheets or a large amount of css for clean hierarchical code. But don’t get me wrong, nesting absolutely everything can cause problems, the two big ones being; media queries are a real pain, the css becomes hard to maintain (especially among multiple developers).

Variables and mixins can be thought of as a little dose of JS put into the SCSS syntax. Variables are a given, but combine that with mixin’s and we can have multiple css declarations apply to a aforementioned variable. Okay, that's pretty sweet right… but there are a few problems in my eyes with this. Sass is meant to give ‘Syntactically awesome style sheets’, which it does but i think we can all agree a major intended use is to save time. The setup time for sass plus creating the mixins / variables doesn’t seem like it would outway regular css. And if it does we’re talking seconds to minutes, which leads me to think that this idea that sass saves time is a gimmick. But again, that's not to say sass isn’t useful, it’s incredibly useful for; calculations, importing multiple stylesheets, clean hierarchies and such. In short, whether you should use sass or not depends on what you’re creating. An entire website with a large quantity of information? Yea I’d recommend sass. A blog / personal page / lightweight website? Probably just stick with regular css.

PROS

  • Speeds up stylings
  • Good documentation
  • Easy to learn & setup
  • Easy to add to tech stack

CONS

  • Debugging is harder
  • Possibly slower performance
  • Maintainence takes longer
  • Hard to take away from stack after use

  • AUG 2018

    MATERIALIZE CSS

Working with this framework has been, overall, really fun. There are certainly many benefits to using a well known design schema like material and I think subject matter is the key takeaway here. I’d say, if you have a formal subject matter (e.g. banking, commercial software retail, etc.) then definitely consider materialize and that's not to say that informal subject matters shouldn’t, but more so don’t stick to material rules as strictly. Why? Personality can be lost in the mist of perfect spacing, perfect consistency of colors, fonts and so on.

PROS

  • Lightweight (only 29kb)
  • Good documentation
  • Uses Google’s material design
  • Extremely good for mobile development
  • Angular support

CONS

  • Deviation from Material UI voids materializes purpose
  • Not supported on older browsers

Below you’ll see a site I whipped up to showcase some of materializes features. To give context, osu is a rhythm game and this site ‘showcases’ some historical scores in the game. It works, but in hindsight I’d have created a fake banking site to really give the framework the credit it deserves. For a site i spent an hour on it's not too bad, also fully responsive.
